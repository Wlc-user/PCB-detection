💡 Bootloader 是什么？

🔑 一句话解释

Bootloader = 汽车的"重装系统工具"

就像电脑的 BIOS + 重装系统，Bootloader 让 ECU 可以在不拆芯片的情况下更新程序。

🎯 三大作用

  • 程序更新 — 远程OTA升级，不需要进4S店
  • 数据校准 — 写入标定参数（油耗Map、点火角等）
  • 故障恢复 — 刷成砖了还能救回来

💭 核心原理

ECU 启动时先运行 Bootloader，它检查：

  • 是否有升级请求？（诊断仪发 0x10 切换模式）
  • 应用程序是否完整？（CRC校验）
  • 版本是否匹配？（兼容性检查）

全部OK就跳转到 App 区运行，否则留在 Bootloader 等待刷写。

🧠 ECU 内存分区

🏗️ 为什么分区？

就像电脑分区 C盘/D盘，ECU Flash 也需要分区：

  • 安全隔离 — Bootloader 不能被误删
  • 独立更新 — App 可以单独刷写
  • 数据保护 — 标定数据单独存放
/* Flash 分区配置示例 */ const FlashLayout = { "Bootloader": { "start": 0x0000, "size": 0x2400 }, "Application": { "start": 0x2400, "size": 0x9C00 }, "Data": { "start": 0xC000, "size": 0x4000 } };

🔄 完整刷写流程

📋 10步标准流程

  1. 1. 0x10 03 — 进入扩展会话
  2. 2. 0x3E 80 — 保持会话心跳
  3. 3. 0x27 01/03 — 请求安全种子
  4. 4. 0x27 02/04 — 发送密钥解锁
  5. 5. 0x31 01 02 — 擦除内存例程
  6. 6. 0x34 — 请求下载（地址+大小）
  7. 7. 0x36 — 传输数据块（循环）
  8. 8. 0x37 — 传输完成请求
  9. 9. 0x31 02 02 — CRC校验例程
  10. 10. 0x11 01 — 硬复位 ECU

📡 UDS 诊断协议

🔤 UDS = 统一诊断服务

ISO 14229 标准，定义了一套"问答题"：

/* 标准UDS请求格式 */ CAN Frame: [ ID: 0x7E0, // ECU物理寻址 DLC: 8, Data: [ 0x34, // 服务ID: 请求下载 0x00, // 数据格式标识 0x44, // 地址/长度格式 0x000xx2400, // 内存起始地址 0x000x8000, // 内存大小 0x00, 0x00 ] ]
/* 标准UDS响应格式 */ CAN Frame: [ ID: 0x7E8, // ECU响应ID DLC: 8, Data: [ 0x74, // 0x34+0x40 = 肯定响应 0x20, // 块大小(32) 0x00, // 块编号 ... ] ]

📚 Bootloader 知识总结

🔑 核心概念

Bootloader 是 ECU 的"重装系统工具"，固化在 Flash 起始区域，负责应用程序的升级更新。

🔐 安全机制

签名验证 + 安全解锁(0x27) + CRC校验 + 分区保护，确保只有合法固件才能被写入。

📡 UDS协议

0x34/0x36/0x37 三步完成刷写，配合 0x10 会话控制、0x27 安全访问、0x31 例程控制。

🧪 测试重点

刷写中断恢复、版本兼容性、签名伪造抵抗、会话超时处理是四大核心测试场景。
